# 단축키 설정 UX 개선 설계

작성일: 2026-05-30 · 대상 버전: v1.4.0(예정) · 상태: 설계 승인 대기

## 1. 배경 / 문제

"단축키 설정이 어렵다"는 사용자 제보. 코드 분석 결과 세 가지 근본 원인:

1. **타이핑 입력** — 설정창에서 `Ctrl+Shift+G` 같은 문자열을 **직접 타이핑**해야 함(`SettingsWindow.xaml`의 `TxtHk*` TextBox 11개). 형식을 알아야 하고 오타가 남.
2. **조용한 충돌 실패** — `HotkeyManager.Register()`가 Win32 `RegisterHotKey`로 등록. 다른 앱이 그 조합을 선점하면 등록이 **실패(-1)하는데 그 반환값을 버림**(`MainWindow.RegisterHotkeys`). 사용자는 영문도 모르고 "단축키가 안 먹는다"고 겪음.
3. **너무 많음** — 전역 단축키 11개가 한 평면 목록에 나열되어 한눈에 안 들어옴.

## 2. 목표 / 비목표

**목표**
- 키 조합을 **눌러서** 설정(타이핑 제거).
- 충돌을 **즉시 시각적으로** 알리고 빈 조합을 추천.
- 시작 시 등록 실패한 단축키를 **조용히 죽이지 말고** 사용자에게 알림.
- 목록을 **자주 쓰는 / 고급**으로 그룹화해 한눈에.

**비목표 (이번 범위 밖)**
- 저수준 키보드 후크를 통한 **강제 우선권**(다른 앱 이기기) — 백신 오탐·복잡도 때문에 채택 안 함. "감지+안내" 방식으로 결정됨.
- 기본 단축키 개수/기본값 변경 — 기존 사용자 단축키가 깨지므로 **마이그레이션 없음**. 군더더기는 '접기'로 숨김.
- OCR 언어/전처리 설정 재배치(현재 단축키 탭 하단에 있음) — 그대로 둠.

## 3. 설계 개요

네 부분으로 구성. 백엔드 단축키 문자열 포맷(`"Ctrl+Shift+G"`)은 그대로 유지 → `HotkeyManager.ParseHotkeyString`/`Register` 변경 최소화.

### 3.1 '키 누르기' 입력 컨트롤 (`HotkeyCaptureBox`)
- 읽기전용 표시 박스. 클릭하면 "키를 누르세요…" 상태.
- `PreviewKeyDown`으로 조합 캡쳐 → 수정자(Ctrl/Shift/Alt) + 일반 키 한 개.
- **Esc** = 취소(이전 값 복원), **Delete/Backspace** = 비우기(미할당), **✕ 버튼** = 비우기.
- 수정자만 눌린 상태는 무시하고 일반 키가 올 때까지 대기.
- 출력 문자열은 `ParseHotkeyString`이 그대로 해석 가능한 정규형(`Ctrl+Alt+Shift+Key` 순서).

### 3.2 실시간 충돌 감지 + 추천
조합을 캡쳐한 즉시 두 단계 검사:
- **내부 충돌** — 신캡쳐의 다른 단축키와 겹침. 메모리 내 비교(즉시). → 🟡 "전체화면 캡쳐와 겹쳐요".
- **외부/OS 충돌** — 다른 앱/Windows가 선점. `HotkeyManager.IsAvailable()`가 프로브 등록을 시도해 확인. → 🔴 "다른 프로그램이 사용 중" + **추천 조합 버튼**(`Ctrl+Alt+G` 등 빈 조합 자동 탐색).
- 깨끗한 프로브를 위해 **설정창이 열려 있는 동안 전역 단축키를 일시 해제**(`UnregisterAll`)하고, 닫을 때 재등록. 그래야 자기 자신과의 충돌로 오탐하지 않음.

### 3.3 시작 시 등록 실패 알림
- `MainWindow.RegisterHotkeys`가 실패한 등록(반환 -1)을 액션 라벨과 함께 수집.
- 하나라도 실패하면 **트레이 풍선 알림** 1회: "창 캡쳐(Ctrl+Shift+W)가 다른 프로그램과 겹쳐 비활성됐어요. 설정에서 변경하세요."
- 실패 집합을 설정창에 전달 → 해당 행에 🔴 배지 선표시.

### 3.4 목록 그룹화 + 기본값 복원
- **기본(펼침)**: 영역지정(+보조), 전체화면, 창 캡쳐, 스크롤, **스마트 컷**
- **고급(`Expander`, 접힘)**: 자유형, 단위영역, 지정사이즈, 텍스트 캡쳐, 텍스트+번역
- 11개 전부 동작 유지. 하단에 **"기본값으로 복원"** 버튼.

## 4. 컴포넌트 설계 (격리)

순수 로직(테스트 용이) ↔ WPF/Win32(얇게) 분리.

| 단위 | 역할 | 의존 |
|---|---|---|
| `HotkeyInput` (static, 신규) | (수정자,키) → 정규 문자열 포맷; 전역 단축키 유효성 검사 | 없음(순수) |
| `HotkeyConflicts` (static, 신규) | 내부 충돌 탐지; 빈 조합 추천(OS 가용성은 주입된 프로브 함수로) | 없음(순수) |
| `HotkeyCaptureBox` (UserControl, 신규) | 키 캡쳐 UI, `Value`/`ValueChanged` 노출 | WPF, `HotkeyInput` |
| `HotkeyManager` (수정) | `IsAvailable(hotkey)` 프로브 추가; `Register`가 성공/실패 구분 결과 반환 | Win32 |
| `SettingsWindow` (수정) | TextBox→CaptureBox 교체, 그룹화, 배지/추천/복원 배선 | WPF, 위 전부 |
| `MainWindow` (수정) | 등록 실패 수집 + 트레이 알림; 설정 닫힐 때 재등록 | WPF |

**유효성 규칙(`HotkeyInput.IsValidGlobalHotkey`)**
- 일반 키(문자/숫자)는 수정자 ≥ 1 필요(맨 키만 전역 등록 방지).
- PrintScreen, F1–F24는 수정자 없이 허용.
- 수정자만 있고 일반 키 없으면 무효.

## 6. 데이터 흐름

1. 설정창 열림 → `MainWindow`가 `HotkeyManager.UnregisterAll()` → 현재 바인딩 로드 → 시작 실패 집합으로 초기 배지 계산.
2. 사용자가 행 클릭 → 키 입력 → `HotkeyCaptureBox`가 `HotkeyInput.Format`으로 문자열화 → `ValueChanged`.
3. `SettingsWindow`가 `HotkeyConflicts.FindInternalConflict`(즉시) → 없으면 `HotkeyManager.IsAvailable`(OS 프로브) → 배지 갱신, 🔴면 `SuggestAlternative` 제시.
4. 저장 → `_settings.Hotkeys.*`에 기록 → 저장 → `MainWindow`가 설정에서 재등록.
5. 취소 → 변경 폐기 → `MainWindow` 재등록(기존 값).

## 7. 엣지 케이스 / 에러 처리

- **현재 값 재입력**: 편집 중인 항목에 동일 조합 재입력 → 충돌 아님(자기 자신).
- **빈 값**: 보조 영역지정처럼 비우면 미등록(현 동작 유지).
- **프로브 실패해도 실사용 시 충돌 가능**: 프로브 성공이 미래 무충돌을 보장하진 않음(다른 앱이 나중에 선점 가능) → 그래서 시작 시 실패 알림(3.3)이 안전망.
- **추천 탐색 한계**: 수정자 조합을 모두 시도해도 빈 게 없으면(희박) "빈 조합을 못 찾음, 직접 지정" 안내.
- **설정창 열려 있는 동안 전역 단축키 비활성** — 의도된 동작(설정 중). 닫으면 복원.

## 8. 테스트 전략

**단위(xUnit, 신규)**
- `HotkeyInputTests`: 정규형 포맷(`Shift+Ctrl+G`→`Ctrl+Shift+G`), 유효성 규칙(맨 문자 거부, PrintScreen 허용, 수정자만 거부).
- `HotkeyConflictsTests`: 내부 충돌 탐지(정규화 비교), 추천 생성(가짜 OS 프로브 주입 → 점유 집합 피해 빈 조합 반환).

**기존 유지**: `HotkeyManagerTests`(`ParseHotkeyString`) — 포맷 호환 보장.

**수동 검증(체크리스트 문서화)**: 키 캡쳐 UX, OS 프로브 실제 충돌, 트레이 알림, 그룹 접기/펼치기, 기본값 복원.

## 9. 마이그레이션

**없음.** 기본값·문자열 포맷·등록 경로 모두 유지. 기존 `settings.json`의 `Hotkeys.*` 그대로 로드됨. UI 표현만 변경.

## 10. 범위 밖(재확인)

저수준 후크 강제 우선권 / 기본 단축키 축소 / OCR 설정 재배치 — 모두 제외. 필요 시 별도 작업.
